# Web UI (Plan 3) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** A single-page web UI, embedded in the binary and served by axum, that shows the live status board (worst-first, expandable per-component) and lets the user add / edit / delete / run monitors via schema-driven forms — turning the existing JSON API into a usable dashboard + management surface.

**Architecture:** A Preact + Vite + TypeScript app under `ui/`, built to `ui/dist/` and embedded into the Rust binary with `rust-embed`; axum serves it via a fallback route (API routes win; everything else serves the SPA). The UI drives the existing `/api/v1/*` endpoints — no new API surface except a `secret` flag added to the config-field schema so token inputs render masked. Single binary + one container preserved; the Docker build gains a Node stage.

**Tech Stack:** Rust (axum, rust-embed), Preact + Vite + TypeScript. Node only at build time.

## Global Constraints

- **TypeScript, not JavaScript**, for all UI code (user requirement). `.ts`/`.tsx` only.
- Only capitalize the first letter of multi-letter acronyms — `ApiClient`, `HttpCheck`, not `APIClient`/`HTTPCheck` — in both Rust and TS.
- Add Rust deps with `cargo add`; Node deps via the `ui/package.json` the plan specifies (pinned by `npm install` → `package-lock.json`).
- Format Rust with `cargo +nightly fmt`. Keep TS consistent (Vite/tsc default).
- No auth (LAN / behind reverse proxy), consistent with the rest of the app.
- API JSON contracts are fixed (see below) — the UI must match them exactly.

**API contracts the UI consumes (from the running backend):**
- `GET /api/v1/status` → `MonitorStatus[]`: `{ id, name, type_id, config, interval_secs, enabled, status: "ok"|"degraded"|"critical"|null, message: string|null, components: {name,status,critical,message}[], updated_at: string|null }` (monitor fields are flattened in).
- `GET /api/v1/check-types` → `{ type_id: string, schema: { fields: Field[] } }[]`, where `Field = { name, kind: "string"|"int"|"float"|"bool", required: bool, default: any|null, help: string, secret: bool }`.
- `GET /api/v1/monitors`, `POST` (201), `PUT /{id}` (200/404), `DELETE /{id}` (204/404). POST/PUT body: `{ name, type_id, config, interval_secs, enabled? }`.
- `POST /api/v1/monitors/{id}/run` → `CheckReport` `{ status, message, components }`.

- [ ] **Step 2: Write `src/ui.rs`**

Create `src/ui.rs`:
```rust
use axum::http::{header, StatusCode, Uri};
use axum::response::{Html, IntoResponse, Response};

#[derive(rust_embed::RustEmbed)]
#[folder = "ui/dist/"]
struct Assets;

const FALLBACK_HTML: &str =
    "<!doctype html><title>homelab-health</title><h1>homelab-health</h1>\
     <p>UI not built. Run <code>npm --prefix ui run build</code>, or use the JSON API at <code>/api/v1/status</code>.</p>";

/// Serve an embedded static asset; fall back to index.html for SPA routes,
/// and to a minimal inline page when the UI has not been built.
pub async fn serve_asset(uri: Uri) -> Response {
    let path = uri.path().trim_start_matches('/');
    if let Some(content) = Assets::get(path) {
        let mime = mime_guess::from_path(path).first_or_octet_stream();
        return ([(header::CONTENT_TYPE, mime.as_ref())], content.data).into_response();
    }
    // SPA fallback: serve index.html for client-side routes.
    match Assets::get("index.html") {
        Some(index) => Html(index.data.to_vec()).into_response(),
        None => (StatusCode::OK, Html(FALLBACK_HTML)).into_response(),
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[tokio::test]
    async fn serves_fallback_when_ui_absent_returns_html() {
        // In debug/test builds rust-embed reads ui/dist from disk; if it's not
        // built, we still return a 200 HTML page (never a 500/panic).
        let resp = serve_asset("/".parse().unwrap()).await;
        assert_eq!(resp.status(), StatusCode::OK);
    }
}
```

## Web UI

The UI (Preact + TypeScript, in `ui/`) is embedded in the binary via rust-embed

  `npm --prefix ui run dev` — Vite serves on :5173 and proxies `/api` to :8080.

## Verification (controller-run, after the tasks)

Backend tests (`cargo test`) cover the secret flag and the no-panic fallback. The UI itself is verified end-to-end with the real binary and a browser:

1. `npm --prefix ui run build && cargo build` — UI embeds.
2. Run the binary on a test port + temp DB; `curl /` returns the built UI's HTML (not the fallback), `curl /api/v1/status` returns JSON.
3. **Playwright** (available via MCP): load the UI, assert the status board renders the seeded monitors, open "Add monitor", pick a type, confirm the form renders the schema fields (and the MA `token` field is a password input), submit, and assert the new monitor appears on the board.
4. `docker build` + container smoke: `curl /` returns the built UI.

## What this delivers

A usable dashboard + management UI embedded in the single binary: live worst-first status board with per-component drill-down, and schema-driven add/edit/delete/run — served at `/` on the same port as the API, deployed as the same one container. Deferred: the history/timeline view (needs Plan 2b's status-history table) and auth.
